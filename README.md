## Пункт 1: Модульное тестирование разложения arctg(x) в степенной ряд

### Что реализовано:

**Класс:** `lab.pavel.math.ArctgSeries`

**Формула:** Ряд Маклорена для arctg(x):
```
arctg(x) = x - x³/3 + x⁵/5 - x⁷/7 + ... = Σ(n=0 to ∞) [(-1)ⁿ · x^(2n+1)] / (2n+1)
```

**Особенности реализации:**
- Валидация входных данных: `x ∈ [-1, 1]`, `terms > 0`
- Для `x = ±1` возвращается табличное значение `±π/4` (без использования `Math.atan`)
- Оптимизация: накапливаем `power = x^(2n+1)` итеративно, умножая на `x²` каждую итерацию

### Тесты:

**Файл:** `ArctgSeriesTest.java`

**Покрытие:**
- ✅ Параметризованный тест `@ParameterizedTest` с табличными значениями из `@CsvSource`
- ✅ Тесты граничных случаев: `x = 0`, `x = ±1`, `x = ±0.5`, `x = ±0.25`
- ✅ Тесты исключений: отрицательное количество членов, `x` вне диапазона
- ✅ **Важно:** Табличные значения НЕ используют `java.lang.Math` (как требовалось)

**Как показать преподавателю:**
```bash
# Запустить только тесты для пункта 1
.\gradlew.bat test --tests "lab.pavel.math.*"
```

**Что объяснить:**
1. Формула ряда Маклорена и почему она работает только для `|x| ≤ 1`
2. Почему для `x = ±1` используется константа (ряд сходится медленно на границах)
3. Как работает параметризованный тест: один метод тестирует множество значений
4. Точность: `assertEquals(expected, actual, 1e-6)` — погрешность до 10⁻⁶

## Пункт 2: Модульное тестирование алгоритма АВЛ-дерева

### Что реализовано:

**Классы:**
- `lab.pavel.avl.AvlTree` — само АВЛ-дерево
- `lab.pavel.avl.TracePoint` — enum с характерными точками алгоритма

**Характерные точки (TracePoint):**
- `INSERT` — начало вставки
- `GO_LEFT`, `GO_RIGHT` — движение по дереву
- `UPDATE_HEIGHT` — обновление высоты узла
- `REBALANCE` — начало проверки баланса
- `LEFT_LEFT_CASE`, `RIGHT_RIGHT_CASE`, `LEFT_RIGHT_CASE`, `RIGHT_LEFT_CASE` — типы дисбаланса
- `ROTATE_LEFT`, `ROTATE_RIGHT` — вращения
- `DUPLICATE_IGNORED` — игнорирование дубликата
- `SEARCH_HIT`, `SEARCH_MISS` — результаты поиска

**Алгоритм:**
1. Рекурсивная вставка с обновлением высоты
2. Вычисление баланс-фактора: `height(left) - height(right)`
3. При дисбалансе > 1 или < -1 — определение типа и выполнение вращений

### Тесты:

**Файл:** `AvlTreeTest.java`

**Покрытие:**
- ✅ **LL-вращение** (Left-Left): вставка 30 → 20 → 10
- ✅ **RR-вращение** (Right-Right): вставка 10 → 20 → 30
- ✅ **LR-вращение** (Left-Right): вставка 30 → 10 → 20
- ✅ **RL-вращение** (Right-Left): вставка 10 → 30 → 20
- ✅ Игнорирование дубликатов
- ✅ Поиск: переходы влево/вправо, попадание/промах
- ✅ Граничные случаи: пустое дерево

**Как показать преподавателю:**
```bash
# Запустить только тесты для пункта 2
.\gradlew.bat test --tests "lab.pavel.avl.*"
```

**Что объяснить:**
1. **Эталонная последовательность:** Каждый тест содержит `expected` список `TracePoint`, который сравнивается с фактической трассировкой
2. **Пример для LL-вращения:**
   ```
   Вставка: 30 → 20 → 10
   Ожидаемая последовательность:
   INSERT → INSERT → GO_LEFT → UPDATE_HEIGHT → REBALANCE →
   INSERT → GO_LEFT → GO_LEFT → UPDATE_HEIGHT → REBALANCE →
   UPDATE_HEIGHT → REBALANCE → LEFT_LEFT_CASE → ROTATE_RIGHT
   ```
3. **Почему это важно:** Тесты проверяют не только результат (корень дерева), но и **последовательность попадания в характерные точки** алгоритма
4. **Метод `getTrace()`** возвращает неизменяемый список всех точек, через которые прошел алгоритм

## Пункт 3: Доменная модель для заданного текста

### Текст из задания:
> "Толпа раскрыла рты, ослепленная ярким солнцем и молодцеватостью, с которой президентский катер обогнул мыс и вошел в залив. Сияя, он скользил по волнам, описывая широкие дуги. Днище катера не касалось воды, он держался на подушке из ионизированных атомов, но для эффекта он был оснащен ложными стабилизаторами, которые могли опускаться в воду. Они взрезали волны, поднимая их стеной, и оставляли в кильватере пенящуюся борозду, когда катер несся через залив."

### Что реализовано:

**Сущности (классы):**
1. **`Crowd`** (Толпа)
   - Состояния: `astonishmentLevel`, `mouthsOpen`
   - Метод: `reactToSpectacle(brightSun, dashingManeuver)`

2. **`President`** (Президент)
   - Состояние: `onBoard`
   - Метод: `disembark()`

3. **`AtmosphericCutter`** (Катер)
   - Состояния: `driveMode` (IDLE/ION_CUSHION), `inBay`
   - Зависимости: `President`, `Bay`, `Stabilizer`
   - Методы: `passCapeAndEnterBay()`, `switchToIonCushion()`, `lowerFakeStabilizersForEffect()`, `createFoamyWake()`

4. **`Stabilizer`** (Стабилизаторы)
   - Состояние: `lowered`
   - Методы: `lower()`, `raise()`

5. **`Bay`** (Залив)
   - Состояние: `foamyWake`
   - Метод: `setFoamyWake()`

**Важно:** Модель работает **только через состояния**, без вывода в консоль (как требовалось).

### Тесты:

**Файл:** `DomainModelTest.java`

**Покрытие:**
- ✅ Все комбинации состояний толпы: `(false,false)`, `(true,false)`, `(false,true)`, `(true,true)`
- ✅ Полный сценарий: катер входит в залив → включает ионную подушку → опускает стабилизаторы → создает пенящуюся борозду
- ✅ Валидация состояний: нельзя войти в залив без президента, нельзя опустить стабилизаторы без ионного режима, нельзя создать борозду вне залива или с поднятыми стабилизаторами
- ✅ Проверка зависимостей: конструктор не принимает `null`

**Как показать преподавателю:**
```bash
# Запустить только тесты для пункта 3
.\gradlew.bat test --tests "lab.pavel.domain.*"
```

**Что объяснить:**
1. **Доменная модель** — это представление предметной области через классы и их состояния
2. **Состояния меняются** через методы, но модель не выводит текст (в отличие от ЛР3)
3. **Тесты проверяют переходы состояний:**
   - Толпа реагирует на комбинации факторов (солнце + маневр)
   - Катер проходит через последовательность состояний: IDLE → ION_CUSHION → стабилизаторы опущены → борозда создана
4. **Валидация:** Модель защищена от некорректных переходов (например, нельзя создать борозду до входа в залив)

## Проверка покрытия кода (100% coverage)

### Настройка JaCoCo:

В `build.gradle` настроен плагин `jacoco` с требованием **100% покрытия** для всех классов из трех модулей:

```gradle
jacocoTestCoverageVerification {
    violationRules {
        rule {
            element = 'CLASS'
            includes = [
                'lab.pavel.math.*',
                'lab.pavel.avl.*',
                'lab.pavel.domain.*'
            ]
            limit {
                counter = 'LINE'
                value = 'COVEREDRATIO'
                minimum = 1.0  // 100% строк
            }
            limit {
                counter = 'BRANCH'
                value = 'COVEREDRATIO'
                minimum = 1.0  // 100% веток
            }
        }
    }
}
```

### Как проверить покрытие:

```bash
# Запустить все тесты и проверить покрытие
.\gradlew.bat check

# Или только тесты
.\gradlew.bat test

# Посмотреть HTML-отчет о покрытии
# Открыть: build/reports/jacoco/test/html/index.html
```

**Что показать преподавателю:**
1. Запустить `.\gradlew.bat check` — должно быть `BUILD SUCCESSFUL`
2. Открыть `build/reports/jacoco/test/html/index.html` в браузере
3. Показать, что все классы имеют **100% покрытие** (зеленые полоски)
